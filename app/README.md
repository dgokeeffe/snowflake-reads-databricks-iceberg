# Snowflake ↔ Databricks interop demo app

A six-section Databricks App (FastAPI + React/Vite/Tailwind) that walks an
audience through the full interop story:

1. **Iceberg refresh** (Snowflake reads Databricks) — write rows into a UC
   managed Iceberg table, watch Snowflake lag, refresh, watch it catch up.
   Side-by-side latest-rows tables highlight the rows Snowflake can't see yet
   (event_ids are monotonic so the diff is real), plus the network architecture
   diagram of the recommended flow.
2. **Federation setup** — live UC introspection of the Snowflake connection
   (Entra M2M OAuth) and the foreign catalogs built on it.
3. **RBAC personas** (Databricks reads Snowflake) — the identical query through
   two role-pinned foreign catalogs: row access + masking enforced by Snowflake,
   live policy DDL via `GET_DDL`. Prereq: `setup_snowflake_rbac_demo.py` in the
   companion benchmark project, or point `RBAC_CATALOG_*` at your own.
4. **TPC-DS benchmarks** — the access-path comparison chart (generated by
   `scripts/analysis/generate_tpcds_path_comparison.py` in the benchmark
   project) + headline medians.
5. **Metric view over federation** — uncached `MEASURE()` queries against a UC
   metric view sourced from the federated `store_sales` (DDL committed at
   `databricks_queries/metric_views/` in the benchmark project), with its YAML,
   source, and Catalog Explorer / lineage / Genie link-outs.
6. **Genie** — link-out to a Genie space whose data asset is the metric view
   (created by `deploy/create_genie_space.py`).

## Local dev

```bash
# backend (from app/)
uv venv && uv pip install -r requirements.txt
cp ../.env .  # or set SNOWFLAKE_*, DATABRICKS_HOST, DEMO_* vars
uv run uvicorn backend.main:app --reload --port 8000

# frontend (separate shell)
cd frontend && npm install && npm run dev   # http://localhost:5173, proxies /api
```

`DATABRICKS_AUTH_TYPE=azure-cli` + `az login` works locally; the Apps runtime
injects SP credentials in production.

## Deploy as a Databricks App

Prereqs: the Snowflake external volume + OBJECT_STORE catalog integration exist
(repo root README / benchmark-project terraform), and the foreign catalogs for
tabs 2/3/5 exist. Then, from `app/`:

```bash
export DATABRICKS_HOST=https://adb-....azuredatabricks.net
export DATABRICKS_AUTH_TYPE=azure-cli
export DATABRICKS_WAREHOUSE_ID=<warehouse id>

# 1. Build the frontend (output is committed — see gotcha 3)
cd frontend && npm install && npm run build && cd ..

# 2. Secret scope for the Snowflake password
databricks secrets create-scope iceberg-demo
databricks secrets put-secret iceberg-demo snowflake-password

# 3. Create the app WITH resources attached (satisfies app.yaml's
#    `valueFrom: snowflake-password` and warehouse CAN_USE in one step).
#    Edit deploy/create_app.json: set the warehouse id first.
databricks apps create --json @deploy/create_app.json

# 4. Grant the app SP everything the six tabs need (incl. EXTERNAL USE SCHEMA)
uv run python deploy/grant_app_sp.py

# 5. Genie space for tab 6 → put the printed id in app.yaml GENIE_SPACE_ID
uv run python deploy/create_genie_space.py

# 6. Adjust app.yaml env values for your workspace, then ship it
databricks sync . "/Users/<you>/iceberg-refresh-demo" --full
databricks apps deploy iceberg-refresh-demo \
  --source-code-path "/Workspace/Users/<you>/iceberg-refresh-demo"
```

### Deploy gotchas (each one cost us a debugging cycle)

1. **Apps injects `DATABRICKS_HOST` without a scheme.** Anything that
   string-builds URLs from it (PyIceberg UC-REST) fails with "No scheme
   supplied". Normalized at startup in `backend/main.py` — keep that block.
2. **The Apps build venv persists across deployments.** pip will report
   "Requirement already satisfied" for a same-version `git+` package even when
   the pinned SHA changes. Ship fixes in the app's own source files; bump the
   SHA pin in `requirements.txt` for reproducibility, not as a delivery
   mechanism.
3. **`databricks sync` respects `.gitignore`.** The built frontend
   (`backend/static/`) must be committed, not ignored — otherwise it silently
   never reaches the workspace and the deployed app 404s on `/` while `/api/*`
   works.
4. **`EXTERNAL USE SCHEMA` is required** for the PyIceberg metadata read in
   `/api/refresh`. It is not part of `ALL PRIVILEGES` and may need a metastore
   admin. The error it prevents is swallowed into app logs as "No Iceberg
   metadata found".
5. The federation-overview tab lists catalogs via `catalogs.list()`, which only
   returns what the SP can see — hence the visibility-only `USE CATALOG` grants
   in `grant_app_sp.py`.

## Environment variables (app.yaml)

| Var | Default | Meaning |
|---|---|---|
| `DEMO_CATALOG` / `DEMO_SCHEMA` / `DEMO_TABLE` | `main.iceberg_demo.demo_events` | UC Iceberg table the demo writes |
| `DEMO_SNOWFLAKE_DB` / `DEMO_SNOWFLAKE_SCHEMA` | `ICEBERG_DEMO.PUBLIC` | Snowflake side of the table |
| `SNOWFLAKE_EXTERNAL_VOLUME` | `databricks_uniform_volume` | Existing external volume over the UC storage account |
| `SNOWFLAKE_ACCOUNT` / `SNOWFLAKE_USER` / `SNOWFLAKE_PASSWORD` / `SNOWFLAKE_WAREHOUSE` | — | Snowflake connection (password via secret resource) |
| `DATABRICKS_WAREHOUSE_ID` | first warehouse | SQL warehouse for writes/counts/MEASURE |
| `RBAC_CATALOG_GLOBAL` / `RBAC_CATALOG_AU` | `rbac_demo_global` / `rbac_demo_au` | Role-pinned foreign catalogs (tab 3) |
| `FEDERATION_CONNECTION` | `davidokeeffe_snowflake_conn_m2m` | UC connection shown on tab 2 |
| `METRIC_VIEW` | `…default.snowflake_sales_metrics` | Metric view for tab 5 |
| `GENIE_SPACE_ID` | — | Genie space for tab 6 (from `deploy/create_genie_space.py`) |
